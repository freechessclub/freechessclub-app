# Free Chess Club

Free Chess Club is a modern interface for the [Free Internet Chess Server (FICS)](https://freechess.org/), one of the oldest internet chess servers with over 800,000 registered accounts.

You can use the hosted web client at [freechess.club/play](http://www.freechess.club/play). The same codebase also ships cross-platform [Electron](https://www.electronjs.org/) desktop apps through [GitHub Releases](https://github.com/freechessclub/freechessclub-app/releases) and [Capacitor](https://capacitorjs.com/) mobile apps for iOS and Android.

## What is in this repository

This repository contains the TypeScript web client, the static assets used by the client, the Electron desktop wrapper, and the Capacitor mobile projects.

Important directories:

| Path | Purpose |
| --- | --- |
| `src/` | Source HTML, TypeScript, styles, data, images, and service worker source. Most app changes start here. |
| `src/js/` | TypeScript application modules for the board, chat, game state, parser, settings, storage, UI, profiles, tournaments, and platform entry points. |
| `src/assets/` | Source CSS, piece sets, board themes, images, and bundled data such as openings. |
| `dev/` | Development build output generated by webpack. |
| `www/` | Production web build output generated by `npm run bundle`. This is the web app payload used by releases. |
| `app/` | Capacitor web directory. Mobile scripts copy the current `www/` build here before running Capacitor. |
| `dist/` | Electron main/preload build output. |
| `android/` | Capacitor Android project. |
| `scripts/` | Build helper scripts, including mobile asset copying and service worker manifest injection. |

Generated directories such as `dev/`, `www/`, `app/`, `dist/`, Android build output, and packaged release artifacts should generally be treated as build products unless you are intentionally updating release assets.

## Prerequisites

Install the following before working on the project:

- [Node.js](https://nodejs.org/) with npm. The repository declares `npm@10.9.4` in `package.json`.
- Git.
- For desktop builds: the platform tooling required by `electron-builder` for the target operating system.
- For Android builds: Android Studio, an Android SDK, and a configured emulator or device.
- For iOS builds: macOS with Xcode and CocoaPods available through the usual Capacitor/iOS toolchain.

This project uses npm. Use the checked-in `package-lock.json` as the source of truth for dependency versions.

## Install

From the repository root:

```bash
npm ci
```

Use `npm ci` for clean, reproducible installs. Use `npm install` only when you intentionally need to update dependencies and refresh the lockfile.

## Run the web app locally

Start the webpack development server:

```bash
npm start
```

Then open [http://localhost:8080/play](http://localhost:8080/play). The dev server also rewrites `/play` to `play.html`, so [http://localhost:8080/play](http://localhost:8080/play) matches the hosted route.

The local server enables hot reloading for the web client and serves the development build from `dev/`. It also sends the cross-origin isolation headers used by browser features that depend on WebAssembly or worker isolation.

## Common scripts

| Command | What it does |
| --- | --- |
| `npm start` | Runs `webpack-dev-server` on port 8080 for local web development. |
| `npm run dev` | Creates a development webpack build in `dev/`. |
| `npm run bundle` | Creates a production web build in `www/` and injects the Workbox service worker manifest. |
| `npm run lint` | Runs ESLint against the TypeScript source under `src/`. |
| `npm run app` | Compiles and launches the Electron app entry point directly for a quick local desktop run. |
| `npm run pack` | Builds Electron assets and creates an unpacked desktop app with `electron-builder --dir`. |
| `npm run dist` | Builds Electron assets and creates distributable desktop packages for the current platform. |
| `npm run build` | Builds desktop packages for macOS, Windows, and Linux x64/arm64 targets and publishes according to the Electron Builder config. |
| `npm run android` | Copies the current `www/` build into `app/` and runs the Capacitor Android app. |
| `npm run android:sync` | Copies the current `www/` build into `app/` and syncs Capacitor Android native assets. |
| `npm run android:bundle` | Runs the Android Gradle release bundle task. |
| `npm run android:ci` | Builds the production web app, syncs Android, and creates the Android release bundle. |
| `npm run ios` | Copies the current `www/` build into `app/` and runs the Capacitor iOS app. |

## Production web build

Create the production web assets with:

```bash
npm run bundle
```

This writes hashed JavaScript and CSS assets to `www/`, copies static files from `src/`, builds `play.html`, and injects the precache manifest into `www/service-worker.js`.

The webpack production build intentionally excludes some source-only files and theme inputs from the copied output. If you add a new static asset, make sure it lives under `src/` and is either referenced by webpack or copied by the `CopyWebpackPlugin` configuration in `webpack.config.js`.

## Desktop builds

The desktop app is packaged with Electron and Electron Builder. The Electron-specific webpack config builds:

- `src/js/app.ts` as the Electron main process bundle.
- `src/js/preload.js` as the Electron preload bundle.

For a quick packaged local build:

```bash
npm run pack
```

For distributable packages on your current platform:

```bash
npm run dist
```

Full release builds use:

```bash
npm run build
```

That command targets macOS, Windows, and Linux for x64 and arm64 according to the Electron Builder settings in `package.json`.

macOS release builds must be signed with a Developer ID Application certificate and notarized. The GitHub Actions Electron release workflow requires these repository secrets:

- `CSC_LINK`: base64-encoded `.p12` Developer ID Application certificate, or another electron-builder-supported certificate link.
- `CSC_KEY_PASSWORD`: password for the signing certificate.
- `APPLE_API_KEY`: App Store Connect `.p8` API key contents.
- `APPLE_API_KEY_ID`: App Store Connect API key ID.
- `APPLE_API_ISSUER`: App Store Connect issuer ID.

## Mobile builds

Mobile apps use Capacitor. The Capacitor `webDir` is `app/`, and the mobile scripts populate it by copying the current production web build from `www/`.

For local mobile work, build the web app first:

```bash
npm run bundle
```

Then run the platform you need:

```bash
npm run android
npm run ios
```

For Android release automation, use:

```bash
npm run android:ci
```

That command runs the production web build, syncs Capacitor Android, and creates the Android release bundle with Gradle.

## Code quality checks

Before opening a pull request, run:

```bash
npm run lint
npm run bundle
```

There is currently no `npm test` script. Until automated tests are added, `npm run lint`, `npm run bundle`, and a focused manual check in the browser are the basic validation path.

Useful manual checks include:

- Opening `/play` locally and signing in or connecting as expected.
- Starting or observing a game and confirming board interaction still works.
- Sending and receiving chat messages.
- Opening settings and confirming preferences persist after reload.
- Checking the browser console for new errors.

## Development notes

- The app source is TypeScript compiled through webpack and `ts-loader`.
- Browser-facing code starts from `src/js/index.ts`.
- Electron uses separate main and preload entries in `src/js/app.ts` and `src/js/preload.js`.
- The service worker source lives in `src/js/service-worker.js`; production builds emit `www/service-worker.js` and inject a Workbox precache manifest.
- Capacitor mobile builds use `capacitor.config.ts`, with `app/` as the web directory.
- Board and UI styling lives mostly under `src/assets/css/`, with chessboard assets and themes under `src/assets/css/images/`, `src/assets/css/pieces/`, and `src/assets/css/themes/`.

## Contributing

Contributions are welcome. Keep changes focused, prefer the existing TypeScript and webpack patterns, and include enough manual verification detail in your pull request for reviewers to reproduce what you checked.

When changing user-facing behavior, consider all supported surfaces: hosted web, Electron desktop, and Capacitor mobile. A change that works in the browser may still need a quick desktop or mobile smoke test if it touches storage, keyboard input, service workers, native wrappers, or layout-sensitive UI.
